Step-by-step data flow — what happens when you search
1
You type "surjeet singh" and press Enter
The browser is displaying index.html. The search box has onkeydown wired to doSearch() in app.js.
index.html app.js → doSearch()
2
app.js builds the API request
getFilters() reads year range, sort order, page size from the sidebar. doSearch() assembles fetch('/api/search?q=surjeet+singh&year_from=2007&year_to=2026&limit=50&page=1')
app.js → getFilters() + fetch()
3
server.py receives the request
FastAPI routes it to the search() function. has_fts() checks if publications_fts table exists — it does. build_fts_query("surjeet singh") returns "surjeet singh" (phrase search, not OR).
server.py → search() → build_fts_query()
4
SQLite runs the FTS5 query
SELECT p.* FROM publications_fts f JOIN publications p ON p.rowid=f.rowid WHERE publications_fts MATCH '"surjeet singh"' AND p.year BETWEEN 2007 AND 2026 LIMIT 50 OFFSET 0. The FTS5 inverted index jumps directly to ~40 matching rows. Takes ~4ms instead of 3500ms.
research_catalogue.db → publications_fts
5
server.py returns JSON
Returns {"results":[...], "total_count":41, "page":1, "total_pages":1, "elapsed_ms":4.2} back to the browser.
server.py → JSON response
6
app.js renders the results
renderSearchResults() calls buildTable(results) which generates HTML rows for each paper. buildPagination() adds page buttons. The metric cards show "41 papers found · 4ms". All of this is injected into the empty <div id="panel-search"> that was defined in index.html.
app.js → renderSearchResults() → buildTable() index.html → #panel-search (filled)

One-time setup connections
arXiv JSON file research_ingestion.py open() + json.loads() 1.49M raw paper records, one per line

build_orcid_cache.py ORCID public API urllib.request (HTTP) Author name → ORCID ID + profile URL
build_orcid_cache.py research_catalogue.db sqlite3 INSERT OR REPLACE orcid_cache table: author_name, orcid_id, status
Every-session connections (browser request cycle)
Browser server.py HTTP GET / Requests the home page
server.py Browser HTTP 200 + HTML Sends index.html content with no-cache headers
index.html server.py <script src="app.js"> Browser automatically requests GET /app.js

Search request cycle
app.js server.py fetch('/api/search?q=...') Query string, year range, sort, page, page size
server.py research_catalogue.db FTS5 MATCH query via sqlite3 Phrase-matched rows from publications_fts → joined to publications
research_catalogue.db server.py fetchall() → list of Row objects Matching paper records with doi, title, authors, year, categories, arxiv_id
server.py app.js JSON response results[], total_count, page, total_pages, elapsed_ms, engine
app.js index.html #panel-search innerHTML = HTML table + pagination buttons + metric cards injected into empty div
Analytics request cycle (fast path)
app.js server.py fetch('/api/analytics') Year range filter
server.py research_catalogue.db SELECT from summary_by_year + summary_by_category Pre-computed counts — microseconds, not minutes
server.py _cache dict (memory) Python dict store Result cached in RAM — subsequent calls return instantly
Researcher profile cycle
app.js server.py fetch('/api/author_match?name=surjeet') Partial name for disambiguation
server.py research_catalogue.db FTS5 MATCH 'authors:"surjeet"*' All author strings containing the name — parsed into distinct names
app.js server.py (parallel) Promise.all([fetch('/api/researcher'), fetch('/api/orcid_lookup')]) Papers + ORCID fetched simultaneously
server.py research_catalogue.db SELECT from orcid_cache Cached ORCID ID (instant) — falls back to live ORCID API if not cached
